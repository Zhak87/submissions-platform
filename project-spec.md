# ТЗ: Платформа сбора проблем и обращений

## 1. Идея проекта

Веб-приложение с лендингом и формой обращений. Люди из разных сфер (предприниматели, медики, и т.д.) заходят на сайт, узнают о проекте и оставляют описание своей проблемы. Все обращения сохраняются в базе данных.

Долгосрочная цель: накопить массив обращений, анализировать их и на основе этого запускать проекты/продукты, решающие реальные проблемы людей.

## 2. Стек технологий

- **Frontend**: Next.js (React) + TypeScript
- **Стили**: Tailwind CSS
- **База данных**: Supabase (PostgreSQL, бесплатный tier — до 500 МБ, достаточно на старте)
- **Хостинг**: Vercel (бесплатный tier, деплой из Git)
- **i18n**: next-intl или next-i18next — переключение RU / KZ

Почему Supabase: бесплатно, готовое REST/JS API, встроенная админка для просмотра данных без написания бэкенда, легко добавить авторизацию для админ-панели в будущем.

## 3. Структура сайта

### 3.1 Лендинг (главная страница)

Секции:

1. **Hero** — заголовок, короткий слоган, кнопка "Оставить обращение" (скролл к форме)
2. **О проекте** — кто мы, что делаем, зачем собираем обращения
3. **Как это работает** — 3 шага: опишите проблему → мы анализируем → создаём решения
4. **Команда** — блок "у нас есть команда, которая помогает разбираться с проблемами" (фото/аватары опционально)
5. **Форма обращения** (см. п. 4)
6. **Футер** — контакты, переключатель языка, соцсети (если есть)

Дизайн: минималистичный, современный, много воздуха, крупная типографика, 1-2 акцентных цвета, мягкие тени/скругления, адаптивность под мобильные устройства.

### 3.2 Переключатель языка

Кнопка RU / KZ в шапке сайта, доступна на всех страницах. Весь контент лендинга и формы (лейблы, плейсхолдеры, сообщения об ошибках, тексты подтверждения) переведён на оба языка.

## 4. Форма обращения

### 4.1 Поля формы

| Поле | Обязательное | Тип | Комментарий |
|---|---|---|---|
| Имя | Да | текст | ФИО или имя |
| Компания | Нет | текст | если есть |
| Телефон | Да | текст/tel | с маской или простой валидацией |
| Email | Да | email | с валидацией формата |
| Сфера деятельности | Нет (рекомендую добавить) | select | напр.: бизнес, медицина, образование, госсектор, другое — пригодится для будущей аналитики |
| Описание проблемы | Да | textarea | развёрнутый текст |
| Есть ли решение / предложение | Нет | textarea | опциональное поле "если у вас есть решение" |

Рекомендую добавить необязательное поле "Сфера деятельности" — это упростит сегментацию данных при анализе позже, но это опционально и можно убрать, если не нужно.

### 4.2 Поведение

- Валидация на клиенте (обязательные поля, формат email/телефона)
- После отправки — экран/сообщение благодарности (на RU/KZ)
- Защита от спама: простое honeypot-поле или Cloudflare Turnstile (бесплатно)
- Отправленные данные сразу пишутся в Supabase через серверный API-роут (не напрямую с клиента — чтобы не светить ключи)

## 5. Схема базы данных (Supabase / PostgreSQL)

Таблица `submissions`:

| Колонка | Тип | Описание |
|---|---|---|
| id | uuid, PK | генерируется автоматически |
| created_at | timestamp | дата и время подачи, по умолчанию now() |
| name | text | имя заявителя |
| company | text, nullable | название компании |
| phone | text | телефон |
| email | text | email |
| category | text, nullable | сфера деятельности |
| problem_description | text | описание проблемы |
| proposed_solution | text, nullable | предложенное решение |
| language | text | на каком языке отправлена форма (ru/kz) |
| status | text | статус обработки: new / reviewed / in_progress / resolved (для внутреннего использования) |

Индексы: по `created_at` и `category` — пригодится при будущей аналитике.

## 6. Админ-панель / просмотр данных (на первом этапе)

На старте можно не делать отдельную админку — использовать встроенный Table Editor в Supabase (веб-интерфейс, доступен сразу из коробки).

На следующем этапе — отдельная защищённая страница `/admin` с таблицей заявок, фильтрами по статусу/категории/дате, возможностью экспорта в CSV.

## 7. Будущая аналитика (следующие этапы, не входит в MVP)

- Дашборд с количеством обращений по категориям, регионам, динамикой по времени
- Кластеризация похожих проблем (NLP по полю `problem_description`)
- Приоритизация: какие проблемы встречаются чаще всего
- На основе кластеров — формирование бэклога будущих продуктов/проектов

## 8. Этапы разработки (MVP)

1. Настройка Next.js проекта, Tailwind, деплой на Vercel
2. Настройка Supabase-проекта, создание таблицы `submissions`
3. Вёрстка лендинга (все секции п. 3.1) на RU
4. Реализация формы + API-роут записи в БД
5. Добавление переводов, переключатель RU/KZ
6. Валидация, защита от спама, экран благодарности
7. Тестирование на мобильных устройствах
8. Публикация
